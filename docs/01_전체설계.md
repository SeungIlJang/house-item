# 1단계: 전체 설계

> 집안의모든것(home-items) 프로젝트 · 작성일 2026-07-17

---

## 1. 이번 단계의 목표

집을 짓기 전에 **설계도**를 그리는 단계입니다.
- 무엇을 만들지(요구사항·MVP 범위)를 확정하고
- 데이터가 어떻게 흐르는지, 데이터베이스를 어떤 표로 만들지(ERD)
- 어떤 API가 필요한지 목록으로 정리합니다.

이 단계에서는 **실제 애플리케이션 코드를 거의 작성하지 않습니다.** 설계 문서만 남깁니다.

---

## 2. 설계 설명 (핵심 선택 이유)

| 선택 | 이유 |
|------|------|
| **모노레포** (백엔드+프론트 한 폴더) | 초기 MVP는 배포·버전 관리를 함께 하는 게 단순함 |
| **계층 분리** (Router→Service→Repository→DB) | 역할이 나뉘어 테스트·수정이 쉬움. Router에서 DB 직접 호출 금지 |
| **JWT Access Token** | 서버가 세션을 저장하지 않아 확장에 유리. Refresh Token은 MVP 이후 |
| **소유권 검사 필수** | 모든 데이터에 `user_id`를 기준으로 "내 것만" 접근하도록 강제 |
| **수납공간 계층형(parent_id)** | 붙박이장 > 오른쪽 칸 > 두 번째 서랍처럼 깊이 표현 |
| **이미지 = 파일 시스템 + 인터페이스 분리** | 나중에 S3로 갈아끼우기 쉽게. DB에 BLOB 저장 금지 |
| **응답 래퍼 통일** (`success/data/message`) | 프론트가 항상 같은 모양으로 처리 |
| **JSON은 camelCase, 파이썬은 snake_case** | 각 진영의 관례를 지키되 경계에서 변환 |

> 용어 한 줄 설명
> - **JWT**: 로그인 정보를 담아 서명한 토큰. 서버가 별도 저장 없이 신원을 확인.
> - **ORM**: 파이썬 객체와 DB 표를 자동으로 연결해 주는 도구(SQLAlchemy).
> - **마이그레이션**: DB 구조 변경을 코드로 기록·적용하는 것(Alembic).

## 3. MVP 범위 확정

### 이번에 만드는 것 (1차 MVP)
1. 회원가입 / 2. 로그인·로그아웃
3. 집 등록 / 4. 방 등록 / 5. 수납공간 등록(계층형)
6. 물건 등록 / 7. 물건 사진 등록 / 8. 물건 수정·삭제
9. 물건 이름 검색 / 10. 방·수납공간 기준 조회
11. 카테고리·태그 관리
12. 모바일 우선 반응형 UI

### 이번에 만들지 않는 것 (구조만 고려)
- 가족 공동 관리, QR/바코드, 위치 변경 이력
- 카메라 등록, OCR, AI 이미지 분석/자동 분류, 자연어 검색
- 푸시 알림, Android/iOS 네이티브, PWA 오프라인
> "구조만 고려" = 나중에 붙일 수 있게 설계는 열어두되 **지금 구현하지 않음**.

## 5. 데이터 흐름 예시 (물건 등록)

```text
1. 사용자가 등록 화면에서 물건 정보 입력 + 사진 선택
2. 프론트 → POST /api/v1/items (JSON, JWT 헤더 포함)
3. Router: 요청 검증(Pydantic), 현재 사용자 확인(get_current_user)
4. Service: 방/수납공간/카테고리가 "내 것"인지 소유권 검사 → 업무 규칙 적용
5. Repository: DB에 Item 저장 (트랜잭션)
6. (이미지) 프론트 → POST /api/v1/items/{id}/images (multipart)
7. StorageService(Local)가 파일 저장 → 경로를 ItemImage에 기록
8. 응답: { success: true, data: {...} } (camelCase)
```

## 6. ERD 초안

### 6.1 개체 관계 (텍스트 다이어그램)

```text
User (1) ──< Home (1) ──< Room (1) ──< StorageLocation ──┐ (자기참조 parent_id)
  │                                        ▲              │
  │                                        └──────────────┘
  ├──< Category
  ├──< Tag
  └──< Item >── (N:M) ── Tag         (연결표: ItemTag)
        │
        ├── Home / Room / StorageLocation / Category  (각각 N:1, 참조)
        └──< ItemImage
```

관계 요약
- User 1 : N Home, Category, Tag, Item
- Home 1 : N Room / Room 1 : N StorageLocation
- StorageLocation 1 : N StorageLocation (self, `parent_id`)
- Item N : 1 Home/Room/StorageLocation/Category (nullable 정책은 아래)
- Item N : M Tag (`item_tags` 연결표)
- Item 1 : N ItemImage

### 6.2 테이블 정의

**users**
| 컬럼 | 타입 | 제약 |
|------|------|------|
| id | bigint PK | auto |
| email | varchar | unique, not null |
| password_hash | varchar | not null |
| name | varchar | not null |
| created_at / updated_at | timestamptz | 공통 |

**homes**
| id PK · user_id FK(users) not null · name not null · description · created_at · updated_at |

**rooms**
| id PK · home_id FK(homes) not null · name not null · description · sort_order int default 0 · created_at · updated_at |

**storage_locations**
| id PK · room_id FK(rooms) not null · parent_id FK(storage_locations) nullable · name not null · description · sort_order int default 0 · created_at · updated_at |
- 순환 참조 방지: 자신/자손을 parent로 지정 금지 (8단계에서 검증 로직)

**categories**
| id PK · user_id FK(users) not null · name not null · created_at · updated_at |
- (user_id, name) unique

**tags**
| id PK · user_id FK(users) not null · name not null · created_at · updated_at |
- (user_id, name) unique

**items**
| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | bigint PK | |
| user_id | FK(users) not null | 소유자 |
| home_id | FK(homes) not null | |
| room_id | FK(rooms) nullable | 미지정 허용 |
| storage_location_id | FK(storage_locations) nullable | 미지정 허용 |
| category_id | FK(categories) nullable | |
| name | varchar not null | 검색 대상 |
| description | text | 검색 대상 |
| quantity | int default 1 | |
| memo | text | 검색 대상 |
| purchase_date | date nullable | |
| expiration_date | date nullable | |
| created_at / updated_at | timestamptz | |

**item_images**
| id PK · item_id FK(items) not null · image_url varchar not null · original_filename · sort_order int default 0 · created_at |
- 대표 이미지 = sort_order 최솟값(0)

**item_tags** (연결표)
| item_id FK(items) · tag_id FK(tags) · PK(item_id, tag_id) |

> nullable 정책: `home_id`는 필수(물건은 반드시 어느 집에 속함), `room_id`·`storage_location_id`·`category_id`는 선택. 삭제 정책은 각 CRUD 단계에서 확정(기본 방향: 상위 삭제 시 하위 CASCADE 또는 차단 — 8·10단계에서 결정).

## 7. API 목록 (v1)

기본 경로: `/api/v1` · 인증: `Authorization: Bearer <token>` (auth 제외)

### 인증 / 사용자
```text
POST   /auth/signup            회원가입
POST   /auth/login             로그인 → access token
GET    /users/me               내 정보
```

### 집 / 방
```text
GET    /homes                  내 집 목록
POST   /homes                  집 생성
GET    /homes/{home_id}        집 상세
PUT    /homes/{home_id}        집 수정
DELETE /homes/{home_id}        집 삭제
GET    /homes/{home_id}/rooms  방 목록
POST   /homes/{home_id}/rooms  방 생성
GET    /rooms/{room_id}        방 상세
PUT    /rooms/{room_id}        방 수정
DELETE /rooms/{room_id}        방 삭제
```

### 수납공간 (계층형)
```text
GET    /rooms/{room_id}/storage-locations      트리 조회
POST   /rooms/{room_id}/storage-locations      생성 (parent_id 선택)
GET    /storage-locations/{id}                 상세 (전체 경로 포함)
PUT    /storage-locations/{id}                 수정
DELETE /storage-locations/{id}                 삭제
```

### 카테고리 / 태그
```text
GET    /categories   POST /categories   PUT /categories/{id}   DELETE /categories/{id}
GET    /tags         POST /tags         PUT /tags/{id}         DELETE /tags/{id}
```

### 물건
```text
GET    /items                  목록 (페이지)
POST   /items                  생성
GET    /items/{item_id}        상세 (보관 위치 경로 포함)
PUT    /items/{item_id}        수정
DELETE /items/{item_id}        삭제
GET    /items/search           검색 (아래 조건 조합)
POST   /items/{item_id}/images 이미지 업로드 (multipart)
DELETE /items/{item_id}/images/{image_id}  이미지 삭제
```

### 검색 조건 (조합 가능)
```text
keyword, home_id, room_id, storage_location_id, category_id, tag_id
page, size, sort
예) GET /items/search?keyword=충전기&room_id=1&category_id=2&page=1&size=20
```

### 상태 확인
```text
GET    /health                 서버 상태 (인증 불필요)
```

## 8. 응답 형식 (공통)

성공
```json
{ "success": true, "data": { "id": 1, "name": "휴대전화 충전기" }, "message": null }
```
오류
```json
{ "success": false, "data": null, "message": "해당 물건을 찾을 수 없습니다.", "errorCode": "ITEM_NOT_FOUND" }
```
목록(페이지)
```json
{ "success": true,
  "data": { "content": [], "page": 1, "size": 20, "totalElements": 0, "totalPages": 0 },
  "message": null }
```
- JSON 필드는 **camelCase**, 파이썬 내부는 snake_case.
